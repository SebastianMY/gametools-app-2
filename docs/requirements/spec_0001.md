# Functional Specification Document: Game Companion Mobile App

## 1. Project Summary

**Project Name:** Game Companion Mobile App

**Description:** A cross-platform mobile application for iOS and Android that provides essential tools for board game and card game players. The application replaces physical game components—dice, score tracking sheets, and player selection methods—through an intuitive, visually engaging interface inspired by a real game table. The app operates entirely offline with local device storage, requiring no backend infrastructure, user accounts, or internet connectivity.

**Target Users:** Players of all ages participating in card games and board games who need quick, reliable game management tools.

**Platform:** iOS and Android via React Native and Expo.

---

## 2. Objectives and Scope

### 2.1 Primary Objectives

1. **Replace Physical Game Components:** Provide digital substitutes for dice, score tracking, and player selection mechanisms commonly used in tabletop gaming.
2. **Enable Seamless Game Resumption:** Allow players to save and restore game state automatically across app sessions without user intervention.
3. **Create an Intuitive User Experience:** Design an interface that is immediately usable by players of varying ages and technical proficiency without tutorial or documentation.
4. **Maintain Complete Offline Functionality:** Ensure the application operates entirely without internet connectivity or backend services.

### 2.2 Scope Boundaries

**In Scope:**
- Dice rolling functionality with 1–6 dice selection and visual animations
- Score management for up to 8 players per game session
- Multi-touch-based random player selection (lottery/draw mechanism)
- Automatic local persistence of game state
- Support for multiple concurrent saved games
- Visual design inspired by real game table aesthetics
- Full WCAG AA accessibility compliance

**Out of Scope:**
- User authentication or account management (v1)
- Cross-device or cloud synchronization
- Specific game rule implementations (Poker, Truco, UNO, etc.)
- Online multiplayer gameplay
- In-app purchases, premium features, or advertising
- Game-specific scorekeeping rules or special mechanics

## 3. System Actors

### 3.1 Primary Actor

**Anonymous User**
- **Description:** Any person who opens the application without creating an account or providing login credentials.
- **Interactions:** 
  - Accesses all features directly upon app launch
  - Creates, manages, and resumes game sessions
  - No persistent user profile or authentication state
  - Can use the app across multiple sessions and devices independently

## 4. Use Cases

### 4.1 Use Case 1: Roll Dice

**Actor:** Anonymous User

**Precondition:** User has opened the Dice section of the application.

**Primary Flow:**
1. User views the dice roller interface.
2. User selects the desired number of dice to roll (1–6) via an on-screen visual control.
3. User taps the "Roll" button.
4. The dice animate with a spinning effect, simulating a roll.
5. The application displays the result of each die individually.
6. The application displays the sum total of all dice.
7. User can tap the "Roll" button again to roll the same number of dice.

**Alternative Flows:**
- **A1: Change Dice Count Before Rolling**
  1. User adjusts the dice count selector before tapping "Roll."
  2. The UI updates to show the new number of dice.
  3. Flow continues from step 3 (Primary Flow).

- **A2: Clear Results**
  1. After viewing results, user taps a "Clear" or "New Roll" button.
  2. The results are cleared and the interface returns to the ready state.
  3. User can proceed with a new roll.

**Postcondition:** Dice results are displayed and remain visible until user initiates a new roll or exits the Dice section.

---

### 4.2 Use Case 2: Create and Manage a Score Game

**Actor:** Anonymous User

**Precondition:** User has opened the Score section of the application.

**Primary Flow:**
1. User views a list of saved games (if any exist).
2. User taps "New Game" to create a new score tracking session.
3. A dialog prompts the user to enter names for each player (up to 8 players).
4. User enters player names and confirms creation.
5. The application initializes a new game with all players at 0 points.
6. The game state is automatically saved to local device storage.
7. The user sees the game interface with player names and score display (0 points each).
8. User taps a "+1", "+5", "+10" button (or similar increment control) next to a player's name to add points.
9. User taps a "−1", "−5", "−10" button (or similar decrement control) next to a player's name to subtract points.
10. The player's score updates immediately in the UI.
11. The updated game state is automatically saved to local device storage.
12. Steps 8–11 repeat as needed throughout the game session.

**Alternative Flows:**
- **A1: Resume a Saved Game**
  1. User views the list of saved games.
  2. User taps on a previously saved game from the list.
  3. The application loads the exact game state, including all player names and scores.
  4. User can continue modifying scores (steps 8–11, Primary Flow).

- **A2: Delete a Saved Game**
  1. User views the list of saved games.
  2. User performs a long-press or swipe gesture on a game entry.
  3. A delete confirmation dialog appears.
  4. User confirms deletion.
  5. The game is removed from the saved games list and local storage.

- **A3: Modify Player Scores with Custom Amounts**
  1. User taps directly on a player's current score (or a dedicated edit button).
  2. A text input field allows the user to enter a specific point value.
  3. User enters the new value and confirms.
  4. The player's score is updated to the specified value.
  5. The game state is automatically saved.

- **A4: Close and Reopen App Without Losing Progress**
  1. User is in an active game session.
  2. User closes the application or locks the device.
  3. Hours, days, or weeks later, user reopens the application.
  4. The application automatically loads the last active game session.
  5. All player names and scores are restored exactly as they were before closing.
  6. User can resume playing without any manual recovery steps.

**Postcondition:** Game state (player names and scores) is persistently stored locally. User can resume the game at any time in the future.

---

### 4.3 Use Case 3: Multi-Touch Player Selection (Lottery)

**Actor:** Anonymous User

**Precondition:** User has opened the Lottery/Selection section of the application.

**Primary Flow:**
1. User sees the Lottery interface with an empty game surface.
2. Each player (typically 2–8) places a finger on the touchscreen simultaneously or in rapid succession.
3. For each detected touch, the application displays a colored circle at that contact point.
4. Each circle is assigned a unique, visually distinct color.
5. The application waits 3 seconds for touches to stabilize (no new touches detected and existing touches remain stable).
6. After 3 seconds, the device vibrates (haptic feedback).
7. The application randomly selects one of the active fingers/circles as the winner.
8. The winner's circle grows with an animation effect to emphasize the selection.
9. All other circles fade out or disappear.
10. The winning player is identified (e.g., "Player in Red circle wins!" or similar visual indication).

**Alternative Flows:**
- **A1: Unstable Touches During Countdown**
  1. User removes a finger or new touches are detected during the 3-second wait.
  2. The countdown resets to 3 seconds.
  3. Flow continues from step 5 (Primary Flow).

- **A2: Single or No Touches**
  1. Only one finger is placed on the screen, or no fingers are placed.
  2. The application displays a message or visual cue indicating a minimum number of participants is required (e.g., 2+ players).
  3. No winner is selected until sufficient touches are detected.

- **A3: Replay Selection**
  1. After a winner is announced, the user can tap a "Play Again" or "New Selection" button.
  2. All circles disappear and the interface resets to the ready state (step 1).
  3. Waiting for new touches begins again.

**Postcondition:** A winning player has been randomly selected and visually highlighted. The result remains visible until the user initiates a new selection or exits the Lottery section.

## 5. Functional Requirements

### 5.1 Dice Module

| ID | Requirement | Description |
|---|---|---|
| FR-D1 | Dice Count Selection | User can select 1 to 6 dice via an on-screen control (e.g., increment/decrement buttons or a number picker). |
| FR-D2 | Roll Animation | When the user taps the "Roll" button, all selected dice animate with a spinning/tumbling effect for 0.5–1.5 seconds. |
| FR-D3 | Individual Die Results | After the animation completes, each die displays its result as a numeric value (1–6) individually. |
| FR-D4 | Total Sum Display | The application displays the sum of all rolled dice prominently. |
| FR-D5 | Multiple Rolls | User can tap the "Roll" button repeatedly to roll the same number of dice without re-selecting the dice count. |
| FR-D6 | Result Persistence | Dice results remain visible until the user initiates a new roll or exits the Dice section. |

### 5.2 Score Module

| ID | Requirement | Description |
|---|---|---|
| FR-S1 | New Game Creation | User can create a new game by providing player names (2–8 players). |
| FR-S2 | Player Name Input | User can input custom names for each player (no character limit specified; assume reasonable max of 50 characters). |
| FR-S3 | Default Score Initialization | All players start with 0 points. |
| FR-S4 | Increment Score | User can add points to a player's score via dedicated buttons (e.g., +1, +5, +10, or custom amount). |
| FR-S5 | Decrement Score | User can subtract points from a player's score via dedicated buttons (e.g., −1, −5, −10, or custom amount). |
| FR-S6 | Custom Score Entry | User can manually enter or modify a player's exact score via a text input mechanism. |
| FR-S7 | Automatic Persistence | Game state (player names and all scores) is automatically saved to local device storage after every score change. |
| FR-S8 | Multiple Saved Games | Multiple independent game sessions can be saved simultaneously on the same device. |
| FR-S9 | Game List Display | User can view a list of all saved games (with dates/times if relevant). |
| FR-S10 | Game Resume | User can select a previously saved game from the list and resume it with all data intact. |
| FR-S11 | Game Deletion | User can delete a saved game from local storage with a confirmation prompt. |
| FR-S12 | Session Restoration | When the app is reopened, the last active game session is automatically restored with all player names and scores. |

### 5.3 Lottery Module

| ID | Requirement | Description |
|---|---|---|
| FR-L1 | Multi-Touch Detection | The application detects and tracks multiple simultaneous finger touches on the screen. |
| FR-L2 | Touch Circle Display | Each detected touch is represented by a colored circle at the contact point. |
| FR-L3 | Unique Color Assignment | Each circle is assigned a distinct color to differentiate between fingers/players. |
| FR-L4 | Touch Stability Monitoring | The application monitors for 3 seconds to ensure touches remain stable (no new touches, no existing touches lost). |
| FR-L5 | Haptic Feedback | After the 3-second stability period, the device vibrates to signal the countdown completion. |
| FR-L6 | Random Winner Selection | One of the active fingers/circles is randomly selected as the winner. |
| FR-L7 | Winner Visualization | The winner's circle grows with an animation effect; all other circles fade out or disappear. |
| FR-L8 | Winner Indication | A clear visual message or indicator shows which player/circle won (e.g., "Red circle wins!"). |
| FR-L9 | Reset on Instability | If a touch is removed or added during the 3-second countdown, the timer resets and continues. |
| FR-L10 | Replay Functionality | User can initiate a new selection after viewing the results without exiting the Lottery section. |

### 5.4 General / Navigation

| ID | Requirement | Description |
|---|---|---|
| FR-G1 | Section Navigation | The application provides clear navigation to three main sections: Dice, Score, and Lottery. |
| FR-G2 | Home/Main Screen | The app displays a main screen (likely with buttons or visual shortcuts) to access each section. |
| FR-G3 | Back Navigation | User can return to the main screen or previous screen from any section. |
| FR-G4 | Intuitive Layout | All interactive elements are organized logically and are immediately understandable without documentation. |

## 7. Technical Constraints

### 7.1 Technology Stack

| Component | Technology | Notes |
|---|---|---|
| **Framework** | React Native | Single codebase for iOS and Android. |
| **Build/Runtime** | Expo | Managed build service (Expo EAS Build), eliminates need for native build configuration. |
| **Local Storage** | AsyncStorage or SQLite | AsyncStorage for simple data (games list metadata); SQLite for structured game state (optional, depending on complexity). |
| **Styling** | React Native StyleSheet or third-party library (e.g., Styled Components) | Cross-platform consistent styling. |
| **Navigation** | React Navigation | Community standard for React Native routing and stack management. |
| **Animations** | React Native Animated API or Reanimated 2 | Smooth 60 FPS animations. |
| **Multi-Touch** | React Native PanResponder or Gesture Handler library | Native multi-touch gesture recognition. |
| **Vibration** | Expo Vibration API | Device haptic feedback. |

### 7.2 Storage Architecture

- **Local Device Storage Only:** No cloud synchronization or remote database.
- **Storage Engine:** AsyncStorage (key-value) or SQLite (relational).
  - AsyncStorage sufficient if game state is simple (JSON serialization).
  - SQLite if game history, statistics, or complex queries are needed in future versions.
- **Data Structure Example:**
  ```
  Game {
    id: UUID,
    name: string (optional),
    createdAt: timestamp,
    lastModified: timestamp,
    players: [
      {
        name: string,
        score: number
      }
    ]
  }
  ```
- **Persistence Trigger:** Automatic save after every score change (debounced to prevent excessive writes).

### 7.3 API and Network

- **No Backend API:** Application does not communicate with any server.
- **No External APIs:** No third-party REST, GraphQL, or WebSocket integrations.
- **No Permissions Required:** No internet permission; optional camera/microphone permissions not needed.

### 7.4 Device Capabilities

- **Multi-Touch Support:** Native support required; no workarounds needed (all modern iOS and Android devices support this).
- **Vibration:** Device vibration API (Expo Vibration) required for haptic feedback in Lottery feature.
- **Touch Precision:** Standard capacitive touch screen; no pressure sensitivity required.

### 7.5 Build and Deployment

| Environment | Details |
|---|---|
| **iOS** | Expo EAS Build for managed CI/CD; App Store submission and review. |
| **Android** | Expo EAS Build for managed APK/AAB generation; Google Play submission and review. |
| **Version Control** | Git; typical branching strategy (main, develop, feature branches). |
| **Code Quality** | ESLint and Prettier for linting and formatting. |
| **Testing** | Jest for unit tests; Detox or similar for end-to-end testing (optional for v1). |

## 9. Assumptions and Dependencies

### 9.1 Assumptions

1. **User Connectivity:** Users always have access to a device (phone or tablet) with a touchscreen; they do not expect real-time sync across multiple devices.
2. **Device Storage:** Devices have sufficient local storage (at least 50 MB available) for the app and saved games.
3. **Touch Input:** All target devices have functional capacitive touch screens; no stylus or special input devices are required.
4. **No Data Sharing Requirement:** Players do not need to share game state electronically; they are co-located and can play on a single device.
5. **Standard Game Rules Knowledge:** Users already know the rules of their games and do not require in-app rule guidance.
6. **No Fraud or Cheat Prevention:** The app does not implement anti-cheating mechanisms (e.g., cryptographic verification of dice rolls); all players are assumed to play fairly.
7. **Language:** App content is in English; users are assumed to understand English or can use device-level translations.
8. **Age of Consent:** Users opening the app are responsible for understanding and following any age restrictions of their games; no parental controls are implemented.

### 9.2 Dependencies

#### External Dependencies
1. **Expo SDK:** Ongoing maintenance and compatibility with React Native and JavaScript/TypeScript.
2. **React Navigation:** Stable community library for routing and navigation.
3. **React Native Animated API:** Part of React Native core; maintained by Meta/React Native community.
4. **Expo Vibration API:** Maintained by Expo; relies on native OS vibration support.
5. **AsyncStorage or SQLite:** Well-maintained libraries with long-term community support.

#### Internal Dependencies
1. **Design System:** A consistent visual design system (colors, typography, spacing) must be established before development.
2. **Accessibility Audit:** Accessibility compliance testing must be performed before release (WCAG AA).
3. **Device Testing:** Functional testing on real iOS and Android devices (both phones and tablets) is required before release.
4. **App Store & Play Store Accounts:** Company or developer accounts needed to publish the app.

#### Developer Skills
- React Native and Expo proficiency.
- JavaScript/TypeScript experience.
- Understanding of mobile UX/accessibility best practices.
- Knowledge of local storage patterns (AsyncStorage, SQLite).
- Experience with animated graphics and gesture handling in React Native.

### 9.3 Open Questions and Clarifications

1. **Score Increment Buttons:** What increment values are most useful (+1, +5, +10, +X)? Should these be customizable per game?
2. **Player Limit Reasoning:** Why 8 players max? This should be validated against typical use cases.
3. **Lottery Color Assignment:** Should colors be pre-assigned in a fixed order (red, blue, green, etc.) or randomly assigned per round?
4. **Game Naming:** Should users name their games (e.g., "Friday Night Poker") or are games identified by timestamp only?
5. **Undo Functionality:** Should users be able to undo the last score change, or is there a limit on undo history?
6. **Data Export:** Should users be able to export game history as CSV or JSON for backup or analysis?
7. **Dice Faces:** Are non-standard dice (d8, d10, d12, d20) needed, or only standard six-sided dice?
8. **Vibration Intensity:** Should vibration intensity be adjustable, or is a single standard vibration sufficient?
